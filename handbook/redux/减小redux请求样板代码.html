    在上一篇中，我们使用redux-saga写了一个较为完整的请求过程，那么在实际使用时我们每次请求都会使用到这一流程， 因此我们自然会想到将这一流程简化。

    # 问题分析

    我们的请求基本只会存在两个结果，对应两个Action Type,参考我们上一篇的代码：

    export const START_FETCH = 'LIST_REQUESTLIST'
export const FETCH_SUCCESS = 'FETCH_SUCCESS'
export const FETCH_FAILED = 'FETCH_FAILED'

const startFetch = (callback) => ({
    type: START_FETCH，
    callback
})
const fetchSuccess = payload => ({
    type: FETCH_SUCCESS,
    payload
})
const fetchFailed = error => ({
    type: FETCH_FAILED,
    error
})

const requestLists = function*({ callback }) {
  try {
    appLoading(); // 展现加载框

    const result = yield handleData.get(DataUrls.searchLists) // 发送网络请求
    if (result && result.success) {
      yield put(fetchSuccess(result.data)); // 请求成功保存数据
      callback && callback();
    } else {
      showModal((result && result.message) || '系统繁忙，请稍后'); 
      yield put(fetchFailed(result.message)); //请求失败错误处理
    }
  } catch (err) {
    yield put(fetchFailed(err)); //错误处理 
    showModal('系统繁忙，请稍后');
  } finally {
    appFinish(); //关闭加载框
  }
};

const Sagas = function* Sagas() {
    yield all([
        takeEvery(START_FETCH, requestLists),
    ]);
};
export default Sagas;

    同时我们需要自热度侧重，添加三个action所对应的状态更改，来相应的对整个请求进行展示。对应的我们需要写如下内容：

    const initState = {
    loading: false,
    failure: false,
    data: {},
    error: null
}

export default funciton FetchReducers(state = initState,action) {
    switch (action.type){
        case START_FETCH:
            return Object.assign({},state,{
                loading: true,
                failure: false,
            })
        case FETCH_SUCCESS:
            return Object.assign({},state,{
                loading: false,
                failure:false,
                data: action.payload
            })
        case FETCH_FAILED:
            return Object.assign({},state,{
                loading:false,
                failure:true,
                error: action.error
            })
    }
}

    针对一个完整的请求，我们往往需要把上述代码全部写一遍，假设我们一个页面有N个请求接口，我们需要把这些近似相同的代码书写无数遍， 显然是很麻烦又不太好的做法，那么我们如何在保证代码流程和可读性的同时，来减少样板代码呢？

    # 封装函数

    针对重复代码，我们第一步想到的就是封装函数，将可变因素作为参数。我们首先需要梳理出这个函数需要做什么：

    1. 获取三个状态的action
    2. 请求过程中，分别对三个action进行处理，并且需要灵活配置请求参数
    3. 自定义的initState，在reducer自动对应三个action状态，更新state

    针对第一步，我们需要在我们的请求中创建三个action，并将其返回出来
